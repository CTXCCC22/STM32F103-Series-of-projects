📑 1. 文件概述
本文件是智能小车系统的主入口与逻辑控制中枢。
其核心设计思想是 MVC 架构（数据、视图与控制解耦）：底层驱动文件（如蓝牙、红外）只负责“搬运数据”，而 main.c 负责统筹所有数据、刷新 OLED 显示（视图UI），并根据当前的系统状态（安全锁）决定是否调用电机驱动（控制器）。

⚙️ 2. 关键全局变量说明
主函数依赖以下几个核心状态变量来维持系统的稳定运转：

run_flag (核心安全锁)：

0：锁定模式（系统正常采集数据并刷新屏幕，但强制切断对电机的指令，确保安全）。

1：运行模式（允许底层动作指令发送至电机）。

bluetooth_mode (模式仲裁)：

0：系统处于“红外循迹”控制模式。

1：系统处于“蓝牙遥控”控制模式（优先级高于红外，收到蓝牙信号即刻切入此模式，按键切回时重置）。

bt_timeout (心跳倒计时)：

配合蓝牙“松手即停”功能使用。每次收到蓝牙字符被赋初值（约10），如果在若干个 while 循环内未收到新字符，倒计时归零，强制执行停车指令。

🔄 3. 主循环 (while) 三段式架构
主循环采用了严格的“三阶段流水线”设计，严禁在阶段间随意跨越或融合代码，以保证状态机的绝对稳定：

🟢 第一阶段：数据采集与 UI 渲染 (不受按键限制)
职责：读取外部世界的所有信息，并立刻显示在 OLED 上。

动作：

调用超声波 Test_Distance() 获取前方障碍物距离。

读取红外循迹传感器的值，并通过查表法转换为字符串。

检查蓝牙中断标志位 bt_rx_flag，若有新指令则提取字符，重置 bt_timeout 心跳，并刷新 OLED 屏幕。

🟡 第二阶段：状态机维护与按键扫描
职责：处理用户的物理交互，维护系统的安全边界。

动作：

扫描物理按键。若按下，则翻转 run_flag 状态。

安全阻断：当 run_flag 从 1 变为 0 的瞬间，立刻调用一次 Car_Stop()，强行刹车。

蓝牙防暴走：只要按下按键，无论如何切换状态，均重置蓝牙接收缓冲区和倒计时。

更新 OLED 第 4 行的电机状态提示（RUN 或 STOP）。

🔴 第三阶段：指令仲裁与动作执行
职责：根据当前的模式和安全状态，决定最终给电机的 PWM 输出。

动作：

首先判断 bluetooth_mode，蓝牙接管具有最高优先级。

通过 switch-case 解析当前的控制字符或红外状态码。

最终门神：所有的电机执行函数（如 Go_Ahead()、Turn_Around_Left()）都被包裹在 if (run_flag == 1) 内。若条件不满足，强制导向 Car_Stop()。

⚠️ 4. 开发与修改注意事项
中断分组定义：本工程在 main 函数的最开头使用了 NVIC_PriorityGroupConfig(NVIC_PriorityGroup_2);。请勿删除，否则可能导致蓝牙中断与超声波冲突死机。

禁止在中断中加延时：所有耗时操作（如屏幕刷新、测距等待）均放在 main.c 的主循环中执行。底层的串口接收中断仅做标志位置位。

添加新动作指令：若需增加如“鸣笛”或“加速”功能：

先在 第一阶段 的 OLED 解析 switch 中添加字符匹配。

再在 第三阶段 的电机执行 switch 中添加硬件调用。
